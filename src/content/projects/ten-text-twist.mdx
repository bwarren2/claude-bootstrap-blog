## The Game

Ten Text Twist is a daily word game (think Wordle's format, Text Twist's mechanics). Each day, players get 10 sets of 6 scrambled letters. Unscramble them to find words. Some sets have multiple valid anagrams — find them all to "perfect" the set. Share your results via an emoji grid:

```
Ten Text Twist #2 — 8/10 ⏱️ 3:42

🟩🟩🟨🟩⬛🟩🟦🟨🟩🟩

bwarren2.github.io/ten-text-twist
```

🟩 solved · 🟨 perfected (all anagrams) · 🟦 solved with hint · ⬛ missed

## The Build: 40 Minutes, 7 Phases

The entire app was built in a single Claude Code session lasting about 40 minutes. Here's how the time broke down:

### Phase 1: Project Setup (0:00–4:00)

Scaffolded a SvelteKit project with `npx sv create`, installed `@sveltejs/adapter-static`, configured the static adapter with GitHub Pages base path, set up prerendering, created `app.css` with a full dark/light theme system using CSS custom properties.

**4 minutes** to go from empty directory to a building, deployable static site skeleton.

### Phase 2: Puzzle Pipeline (10:00–14:00, background through 30:00)

Two Node.js scripts, no external dependencies:

- **`fetch-definitions.js`** — Queries the Free Dictionary API for each 6-letter word, caches results to JSON, handles rate limiting with 150ms delays and 429 backoff, saves progress every 50 words for resume.
- **`generate-puzzles.js`** — Downloads the 12dicts word list, parses 6-letter words (~8100), groups by sorted-letter signature into anagram families (~940 multi-word families), assembles 365 puzzles with a seeded PRNG (mulberry32) for deterministic output.

The definition fetch ran in the background while UI work continued. By the time it was stopped (~550 words cached), there were enough definitions for 365 puzzles with some word reuse.

### Phase 3: Game State (4:00–10:00)

The core of the app lives in four modules:

- **`game.svelte.js`** — Svelte 5 `$state` runes for puzzle data, current set, found words, hint tracking, input buffer, timer, and toast messages. Handles submission validation, auto-advance, set navigation, and localStorage persistence.
- **`settings.svelte.js`** — Theme (dark/light) and mode (timed/untimed) with auto-save.
- **`stats.svelte.js`** — Games played, win rate, streaks, score distribution, best time.
- **`persistence.js`** — Thin localStorage wrapper with a `ttt_` prefix.

Plus four utility modules: anagram validation, puzzle number calculation (days since epoch in UTC), timer formatting, and share text generation.

### Phase 4: UI Components (14:00–22:00)

15 Svelte components built in parallel batches:

**Game components:** `LetterTiles` (6 clickable tiles + shuffle), `InputArea` (6 slots with shake animation), `Keyboard` (on-screen keys showing only the 6 current letters), `SetIndicator` (10 color-coded numbered dots), `HintButton` (definition reveal), `Timer` (countdown), `FoundWords` (words found in current set), `Toast` (feedback messages).

**Chrome:** `Header` (title + icon buttons), `ModeSelect` (timed/untimed selection screen).

**Modals:** `ResultsModal` (emoji grid + answer reveal + share), `StatsModal` (stats + score distribution bar chart), `SettingsModal` (theme/mode toggles), `HelpModal` (rules + scoring legend).

**Orchestrator:** `Game.svelte` ties it all together — keyboard event handling, derived used-letter indices, submission flow, auto-advance on solve.

### Phase 5: Results & Sharing (18:00–22:00)

Built alongside Phase 4. The share flow uses `navigator.share()` with a `navigator.clipboard.writeText()` fallback. The results modal reveals all answers with definitions, color-coded by status.

### Phase 6: Polish (22:00–30:00)

Fixed a11y warnings (added `tabindex` and `onkeydown` to modal dialogs), wrapped module-level `$effect` with `$effect.root()`, fixed the page loader to use SvelteKit's `base` path, added a custom favicon, and generated the full 365-puzzle set.

### Phase 7: Deploy & Iterate (30:00–40:00)

Created the GitHub Actions workflow, initialized the repo, pushed to GitHub. Then three quick UX iterations based on playtesting:

1. **Larger "I'm Done" button** — Increased size, added red hover state to signal finality.
2. **Skip tip** — A contextual hint ("you can skip to the next word and come back any time") that appears after 25 seconds on a set or after finding any word.
3. **Permanent tip dismissal** — Once dismissed, the tip never returns for the rest of the game.

## Architecture

```
ten-text-twist/
  scripts/
    fetch-definitions.js          # Dictionary API → cached JSON
    generate-puzzles.js           # Word list → 365 puzzle files
  static/puzzles/
    001.json ... 365.json         # Pre-generated daily puzzles
    manifest.json                 # Epoch date + count
  src/lib/
    state/                        # Svelte 5 $state singletons
      game.svelte.js              # Core game logic
      settings.svelte.js          # Theme + mode
      stats.svelte.js             # Play statistics
    utils/                        # Pure functions
    components/                   # 15 Svelte components
  src/routes/
    +page.svelte                  # Main page (mode select → game → results)
```

**Key design decisions:**

- **Pre-generated JSON** — Zero runtime API calls. Each puzzle is a static JSON file loaded by day number.
- **UTC puzzle numbering** — All players see the same puzzle on the same calendar day.
- **On-screen keyboard shows only 6 letters** — Prevents invalid input entirely.
- **Timer starts on first input** — Player controls when pressure begins.
- **Free set navigation** — No set is a blocker; skip and return at will.

## Post-Launch Fixes (Day 3)

Three days after launch, a round of playtesting and sharing surfaced five issues — all fixed in a single session.

### The Duplicate Letter Bug

The most interesting bug: when a word has duplicate letters (e.g., two E's at positions 2 and 5), tapping the E at position 5 would disable the E at position 2 instead. The wrong tile grayed out.

**Root cause:** The original code only tracked which *letters* were in the input buffer, not which *tiles* were selected. When `Game.svelte` needed to figure out which display tiles to disable, it reverse-computed indices from letter values using `find()`:

```javascript
// OLD: Game.svelte — always picks the first matching letter
const usedIndices = $derived.by(() => {
    const indices = [];
    const available = game.displayLetters.map((l, i) => ({ letter: l, index: i }));
    for (const inputLetter of game.input) {
        const found = available.find(a => a.letter === inputLetter && !indices.includes(a.index));
        if (found) indices.push(found.index);
    }
    return indices;
});
```

For unique letters this works fine. But `find()` always returns the first match, so tapping the *second* E still disables the *first* E. The information about which tile was actually tapped was lost the moment `addLetter()` received just a letter value.

**The fix:** Track tile indices directly in game state, parallel to the input buffer. `addLetter(letter, sourceIndex)` now accepts an optional tile index — when provided (tile tap), it uses that exact index; when omitted (physical keyboard), it falls back to first-available. The component just reads the authoritative index list instead of reverse-computing it:

```javascript
// NEW: game.svelte.js — tracks exactly which tiles were selected
addLetter(letter, sourceIndex) {
    if (sourceIndex !== undefined && displayLetters[sourceIndex] === upperLetter
        && !inputIndices.includes(sourceIndex)) {
        chosenIndex = sourceIndex;  // use the exact tile that was tapped
    } else {
        chosenIndex = displayLetters.findIndex((l, i) =>
            l === upperLetter && !inputIndices.includes(i));  // keyboard fallback
    }
    input = [...input, upperLetter];
    inputIndices = [...inputIndices, chosenIndex];
}
```

```javascript
// NEW: Game.svelte — just read the authoritative list
const usedIndices = $derived(game.inputIndices);
```

Two files changed, and both `removeLetter()` and `removeLetterAt()` keep the index array in sync so backspace and slot-tapping re-enable the correct tile.

### Mobile Double-Tap Zoom

On mobile Safari and Chrome, rapidly tapping letter tiles triggered the browser's double-tap-to-zoom gesture instead of registering two quick letter inputs. One line in `app.css` fixed it:

```css
*, *::before, *::after {
    touch-action: manipulation;
}
```

`touch-action: manipulation` tells the browser to handle touch events for panning and pinch-zoom only — disabling the 300ms double-tap-to-zoom delay entirely. This is safe for a game UI where double-tap zoom is never the intended interaction.

### Share Link Fixes

Three small fixes to the share text, all in `share.js`:

1. **GitHub Pages URL** — The share link was pointing to the wrong domain. Fixed to `bwarren2.github.io/ten-text-twist`.
2. **Compact format** — Removed blank lines between the header, emoji grid, and link so the share text is tighter when pasted into chat.
3. **HTTPS prefix** — The link was `bwarren2.github.io/...` without `https://`, so Discord and other platforms didn't auto-detect it as a clickable URL. Adding the scheme prefix fixed auto-linkification everywhere.

### Pattern: Playtesting Finds What Code Review Doesn't

All five issues were invisible in the code. The duplicate letter bug passed every mental code review — the `find()` logic *looks* correct. The double-tap zoom only appears on real phone hardware. The share link only breaks when you paste it into Discord and notice it's not clickable.

The lesson: for interactive apps, there's no substitute for actually playing the game on real devices and sharing the results to real platforms. Each bug was found in under a minute of real usage and fixed in under a minute of Claude Code time.
